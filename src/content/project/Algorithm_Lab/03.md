## Question1: Try to proof that sorting algorithms based on ...  
### Ans:  
To determine the minimum height of the decision tree (which corresponds to the minimum number of comparisons required in the worst case), we need to ensure that the tree has at least $n!$ leaves (to account for all possible permutations).
1. Height and Number of Leaves: A binary tree of height $h$ can have at most $2^h$ leaves. Therefore, to have at least $n!$ leaves: $$2^h \ge n!$$ Taking logarithms (base 2) on both sides: $$h \ge \log_2(n!)$$
2. Approximating $\log_2(n!)$: We can use Stirling's approximation for factorials: $$n! \approx \left(\frac{n}{e}\right)^n\sqrt{2\pi n}$$ Taking the logarithm: $$\log_2(n!) \approx n\log_2 n - n\log_2 e + \frac{1}{2}\log_2(2\pi n)$$ The dominant term here is $n\log_2 n$, so: $$\log_2(n!)=\Theta(n\log n)$$ Thus: $$h=\Omega(n\log n)$$ This means that the height of the decision tree (and hence the number of comparisons required in the worst case) must be at least $\Omega(n\log n)$.
3. Any comparison-based sorting algorithm must perform at least $\Omega(n\log n)$ comparisons in the worst case to sort $n$ elements. Therefore, such algorithms cannot have a time complexity faster than $O(n\log n)$.
## Question2: Write down recursive time function and analyze ...  
### Ans:  
$$T(n) = T(n/2) + O(1)$$ Where:
* T(n/2) represents the recursive call on half the array,
* O(1) represents the constant time operations (comparisons and arithmetic)

The Master Theorem applies to recurrences of the form: $T(n) = aT(\frac{n}{b}) + f(n)$
For binary search:
* $a = 1$ (one recursive call)
* $b = 2$ (divides problem in half)
* $f(n) = O(1) = O(n^0)$

Comparing $\log_b(a) = \log_2(1) = 0$ with the exponent of $f(n)$ (which is 0): Case 2 applies where $\log_b(a)$ equals the exponent of $f(n)$, giving: $$T(n) = \Theta(\log n)$$
## Question3: Rewrite above algorithm by while loop.
### Ans:  
```c++
template <typename T>
int binary_search(const T sorted_data[], int n, const T key) {
    int left = 0;
    int right = n - 1;    
    while (left <= right) {
        int mid = left + (right - left) / 2;        
        if (sorted_data[mid] == key) return mid;
        else if (sorted_data[mid] > key) right = mid - 1;
        else left = mid + 1;
    }
    
    return n;
}
```
## Question4: What's the time complexity of above algorihtm.
### Ans:
1. Time Complexity Without Considering $f$:
Each call halves the interval size, so the number of recursive calls (depth of recursion) depends on how many times we can halve the interval before it becomes smaller than $\varepsilon$: $$\frac{b-a}{2^k} \le \varepsilon \Rightarrow k \ge \log_2\left(\frac{b-a}{\varepsilon}\right)$$
So the number of recursive calls is: $$O\left(\log_2\frac{1}{\varepsilon}\right)$$

2. Considering Cost of $f$:
Let the time complexity of evaluating $f(x)$ is $T_f$. Since each recursive step calls $f(x)$ up to 3 times (at a, b, and mid — though a and b can be cached if optimized), the total cost becomes: $$O\left(T_f \cdot \log_2\frac{1}{\varepsilon}\right)$$

3. Final Answer:
The worst-case time complexity of the algorithm is: $$O\left(T_f \cdot \log_2\frac{1}{\varepsilon}\right)$$ Where:
* $T_f$ is the cost of a single function evaluation (depends on what f is).
* $\varepsilon$ is the required precision ($epi$ in code).

## Question5: What is the best $k$ for searching key in $n$ elements?
### Ans:
#### Time Complexity Breakdown:

1. Recursion Depth:  
   In each recursive step, the size of the search space is divided by $k$. Therefore, the recursion depth is the number of times we can divide $n$ by $k$, which is: $$\text{Depth} = O(\log_k n)$$

2. Work Done at Each Level:  
   At each level, the algorithm compares the key with $k-1$ values (to decide which partition to search), so the work done at each level is $O(k)$.

3. Total Time Complexity:  
    The total work is the product of the depth of recursion and the work at each level: $$O(k \cdot \log_k n)$$
   To express this in terms of base 2 logarithms, we use the change of base formula $\log_k n = \frac{\log_2 n}{\log_2 k}$: $$O(k \cdot \frac{\log_2 n}{\log_2 k})$$

#### Optimizing $k$:
We aim to minimize the total time complexity $O(k \cdot \frac{\log_2 n}{\log_2 k})$. To do this, we find the optimal $k$ by differentiating the expression and solving for $k$.
The optimal $k$ that minimizes the time complexity is approximately: $$k \approx \frac{\log_2 n}{\log_2 \log_2 n}$$

#### Conclusion:
The optimal value of $k$ for searching a key in $n$ elements is approximately:
$$k \approx \frac{\log_2 n}{\log_2 \log_2 n}$$
